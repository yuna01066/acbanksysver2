# 거래처 관리

## 기본 정보

- Route: `/recipients`
- 주요 파일: `src/pages/RecipientManagementPage.tsx`
- 주요 사용자: 직원, 관리자
- 화면 목적: 거래처/담당자 정보를 관리하고 견적/프로젝트와 연결한다.

## UI 구조

- 상단: 거래처 제목, 검색, 신규 등록 액션
- 목록: 회사명, 담당자, 연락처, 최근 활동, 문서/프로젝트 연결
- 상세/패널: 거래처 기본 정보, 메모, 문서, 타임라인
- 보조 액션: 파일 업로드, 견적/프로젝트 확인

## 톤앤매너

- 거래처명과 담당자명을 분명히 구분한다.
- 연락처/주소 같은 정보는 읽기 좋게 묶되 과도하게 크게 보이지 않게 한다.
- 타임라인은 업무 기록용으로 차분하게 유지한다.

## 자주 수정하는 요소

- 목록 행/카드 정보 순서
- 거래처 상세 섹션 제목
- 문서 업로드 영역
- 빈 상태와 검색 결과 없음 문구

## 연결 데이터와 주의점

- 연결 데이터: `recipients`, `saved_quotes`, `projects`, `document_files`
- 권한: 고객 정보 접근 권한과 RLS 확인 필요
- 변경 위험: 실제 고객 개인정보를 문서나 테스트 데이터에 노출하지 않는다.

## 정보 위계

- 1순위: 업체명/고객명, 담당자, 연락처
- 2순위: 최근 견적/프로젝트, 메모, 문서 수
- 3순위: 주소, 이메일, 거래 이력
- 낮은 강조: 내부 sync 상태, 오래된 보조 메모

## 상세 설계 기준

- 목록: 업체명과 담당자를 분리하고, 개인 고객은 업체명 fallback을 자연스럽게 처리한다.
- 상세: 기본 정보, 견적/프로젝트, 문서, 메모/타임라인 순서로 업무 활용도가 높은 정보를 먼저 둔다.
- 파일 영역: 업로드/다운로드는 공통 파일 서비스를 기준으로 표현하고 저장소 이름을 직접 노출하지 않는다.
- 검색: 업체명, 담당자명, 전화번호 일부 검색을 고려해 placeholder를 구체적으로 둔다.
- 모바일: 연락처 액션은 버튼으로 넓게 제공하되 개인정보가 과도하게 노출되지 않게 한다.

## 검수 시나리오

- 업체명 없는 개인 고객이 깨진 제목 없이 표시된다.
- 문서가 없는 거래처에서 문서 영역이 빈 상태를 보여준다.
- 견적/프로젝트 연결 링크가 실제 상세 페이지로 이동한다.
- 일반 직원 계정에서 접근 가능한 고객 범위가 정책과 맞는지 확인한다.
