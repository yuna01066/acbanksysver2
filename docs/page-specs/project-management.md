# 프로젝트 관리

## 기본 정보

- Route: `/project-management`
- 주요 파일: `src/pages/ProjectManagementPage.tsx`
- 주요 사용자: 직원, 관리자, 프로젝트 담당자
- 화면 목적: 클라이언트/내부 프로젝트를 검색하고 상태, 입금, 견적, 담당자를 관리한다.

## UI 구조

- 좌측/목록: 프로젝트 검색, 클라이언트/내부 구분, 프로젝트 카드
- 상세: 프로젝트명, 견적 출처, 개요, 타임라인, 수익성, 마일스톤
- 사이드 패널: 단계, 입금 상태, 예상 견적, 생성일, 담당 직원
- 액션: 수정, 삭제, 담당자 배정, 상태 변경

## 톤앤매너

- 프로젝트명과 현재 단계가 가장 먼저 읽혀야 한다.
- 완료/입금/진행 예정 같은 상태는 정확한 의미로 표시한다.
- 긴 프로젝트명은 줄바꿈 또는 truncate 기준을 통일한다.

## 자주 수정하는 요소

- 프로젝트 카드 안의 정보 순서
- 단계/입금 상태 배지
- 상세 탭 구성
- 담당자 배정 UI

## 연결 데이터와 주의점

- 연결 데이터: `projects`, `saved_quotes`, `project_assignments`, `project_milestones`
- 권한: 담당자와 관리자 권한 분리
- 변경 위험: 입금 상태를 완료 상태로 오해하게 만드는 UI 표현을 피한다.

## 정보 위계

- 1순위: 프로젝트명, 단계, 입금 상태, 예상/연결 견적 금액
- 2순위: 거래처, 담당자, 생성일, 납기/마일스톤
- 3순위: 수익성, 내부 문서, 타임라인
- 낮은 강조: 내부 참조 ID, 보조 메타

## 상세 설계 기준

- 목록 카드: 프로젝트명은 가장 굵게, 상태와 입금 여부는 서로 다른 배지 스타일로 구분한다.
- 상세 패널: 단계 변경, 입금 상태, 담당자 배정은 우측 패널에 모아 반복 작업을 빠르게 한다.
- 탭: 개요, 타임라인, 수익성, 마일스톤은 같은 탭 높이와 간격을 유지한다.
- 상태 표현: `완료`는 프로젝트 완료, `입금`은 결제 상태로만 사용한다.
- 모바일: 좌측 목록과 상세를 한 화면에 억지로 넣지 말고 목록 선택 후 상세 전환 구조를 유지한다.

## 검수 시나리오

- 견적서에서 생성된 프로젝트가 "완료"로 오표시되지 않는지 확인한다.
- 입금 상태를 바꿨을 때 목록 카드와 상세 패널이 같은 값으로 갱신된다.
- 담당자 배정 후 담당자 칩이 중복 표시되지 않는다.
- 긴 프로젝트명이 카드와 상세 제목에서 모두 읽히거나 적절히 줄임 처리된다.
